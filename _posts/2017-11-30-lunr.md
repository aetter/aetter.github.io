title: Adding Search Using Lunr.js

One of the features I've always liked about [Sphinx](http://www.sphinx-doc.org/) is that it includes search out of the box.

Years ago, when I first started using Sphinx, I was confused by how the team had implemented the feature. No applications are involved; Sphinx outputs a static site that you can zip and unzip, and the site just works. Did an open source project have a search server somewhere that every single Sphinx site connected to? No, that didn't make any sense. After all, the notion seemed crazy, and search worked whether or not I was connected to the internet. Then I scanned the directory for a finished Sphinx site and saw it: `searchindex.js`.

"Oh," I said, turning to a coworker and laughing. "They build a search index and then drop it into a JavaScript file. The entire file is just a bunch of key-value pairs of strings and the pages in which they appear."

The pros and cons of such an approach are pretty obvious. Because everything happens in the user's web browser, search is completely portable and *should* be extremely responsive. But because the user's browser has to load the entire search index --- as opposed to loading the much smaller set of results that a search server might return --- the search feature will scale poorly and could have severe load times as the index grows.

Then again, if your documentation only has a few dozen pages, your search index might not even exceed 100 KB. Most sites, even for complex products, shouldn't encounter issues. If your company has enough products that search becomes a problem, you can swap out the JavaScript-based search for something more robust.

But all this only applies if you use Sphinx. If you use Jekyll, you're out of luck. You might be tempted to use the Algolia plugin or Google Custom Search. But this choice makes your documentation less portable and might cost you some money. If you distribute the documentation alongside an application, search won't work. Fortunately for this use case, [Lunr.js](https://lunrjs.com/) exists.

Lunr is a JavaScript-based search engine that works similarly to Sphinx's, but has some fantastic features like boosts and wildcards. To add Lunr to a site, you need to do five things:

1. Process your site into a collection of JSON documents.

    The variable should ultimately look like the `documents` variable [here](https://lunrjs.com/guides/getting_started.html). I use the file name `documents.js`. A number of people have used an ingenious Jekyll hack to accomplish this goal without any real code:


    {% raw %}
    ```liquid
    ---
    ---
    {% assign id = 0 %}
    var posts = [
      {% for post in site.posts %}
        {
          "title": "{{ post.title | escape }}",
          "description": "{{ post.description | escape }}",
          "content": "{{ post.content | strip_html | strip_newlines | escape }}",
          "url": "{{ post.url | escape }}",
          "id": "{{id}}"
        }{% unless forloop.last %},{% endunless %}
      {% assign id = id | plus: 1 %}{% endfor %}
    ];
    ```
    {% endraw %}

    If you're not using Jekyll, you can use a library like [Beautiful Soup](https://www.crummy.com/software/BeautifulSoup/bs4/doc/) to process the searchable content of your various HTML tags into JSON. Other methods surely exist.

2. Tell Lunr to [index the documents](https://lunrjs.com/guides/getting_started.html):

    ```javascript
    var idx = lunr(function () {
      this.ref('id')
      this.field('url')
      this.field('title', { boost: 5 })
      this.field('description')
      this.field('content', { boost: 2 })

      posts.forEach(function (post) {
        this.add(post)
      }, this)
    })
    ```

3. Get a search term from the user.
3. Make the search request.
4. Display the results.

    Note: Search results include *references* to any matching documents, *not* the contents of those documents. You still need the original JSON documents in order to display anything beyond a unique ID.

I'm not going to cover these last three steps. Katy DeCorah has a great walkthrough [on her blog](https://katydecorah.com/code/lunr-and-jekyll/), and my code is extremely similar. You can try out the new search feature on the [home page](/).

Ultimately, the `documents.js` file that I build for this site weighs in at 60 KB, which you now download when you visit. Then Lunr has to process it into an index. According to Chrome, the negative impact to page load performance is about a quarter-second:

![Chrome Performance panel](/assets/images/lunr-perf.png)

In terms of clear improvements that I'll probably never implement, the big one is to use web workers to build the search index in a separate JavaScript thread. This way, page load time wouldn't be impacted; search just wouldn't be available for that quarter-second. The Angular team has [a fascinating blog post](http://www.bacondarwin.co.uk/angularjs-docs-performance/) on this very subject. My understanding is that I could also [pre-build the index](https://lunrjs.com/guides/index_prebuilding.html), but I'm unsure if bundling another large JavaScript file with the site would be faster or slower than building the index on the fly.

Regardless, having a tool-agnostic, client-side alternative to the various server-based search engines on the market is a true blessing. While it doesn't fit every use case, Lunr is eminently viable in the field of technical documentation. I'm extremely grateful to [Oliver Nightingale](https://github.com/olivernn) for all his work on the project.
