title: The Lightweight DITA Misnomer
date: 2018-01-03
description: OASIS should rename this deceptive specification.
---

The [Darwin Information Typing Architecture](https://en.wikipedia.org/wiki/Darwin_Information_Typing_Architecture) (DITA) is a large, complex XML-based markup language that a consortium called OASIS maintains. If you're curious about what DITA looks like, Wikipedia has [some code samples](https://en.wikipedia.org/wiki/Darwin_Information_Typing_Architecture#Code_samples).

Realistically, no one should use DITA on a new project under any circumstances. And that's *fine*. Really, it is. Legacy projects still need maintenance and improvements, because the cost and hassle of migrating to more modern systems might not make sense for an organization. I'm glad that OASIS continues to improve and update the DITA specification.

But their new specification, [Lightweight DITA](http://docs.oasis-open.org/dita/LwDITA/v1.0/LwDITA-v1.0.html), is either a) actively deceptive or b) grounded in ignorance as to what "lightweight" means in the context of markup languages. **OASIS should rename it immediately.** I'll explain.

[Lightweight markup languages](https://en.wikipedia.org/wiki/Lightweight_markup_language) aim to provide a simple, unobtrusive syntax that is easy to read in plain text form while still giving structure to a document. The formatting is usually some combination of spacing, symbols, and maybe a few short keywords. These languages arose as a direct response to XML-based markup languages, which are difficult to read, cumbersome to write, and look terrible in side-by-side version control diffs.

I use the following example in Modern Technical Writing to demonstrate the difference between equivalent bits of AsciiDoc, a lightweight markup language, and DocBook, its XML-based sibling.

First, the AsciiDoc:

```
= My Title

This introductory paragraph contains a link to
https://www.google.com[Google].

[source,ruby]
puts "Here is some code."

== My Section

* List item
* Another list item
```

And then the DocBook:

```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE article PUBLIC "-//OASIS//DTD DocBook XML V4.5//EN"
"http://www.oasis-open.org/docbook/xml/4.5/docbookx.dtd">
<?asciidoc-toc?>
<?asciidoc-numbered?>
<article lang="en">
<articleinfo>
    <title>My Title</title>
</articleinfo>
<simpara>This introductory paragraph contains a link to
<ulink url="https://www.google.com">
Google</ulink>.</simpara>
<programlisting language="ruby" linenumbering="unnumbered">
    puts "Here is some code."
</programlisting>
<section id="_my_section">
<title>My Section</title>
<itemizedlist>
<listitem>
<simpara>
List item
</simpara>
</listitem>
<listitem>
<simpara>
Another list item
</simpara>
</listitem>
</itemizedlist>
</section>
</article>
```

When I first heard about so-called Lightweight DITA, I thought, "Oh, how interesting, they came up with an AsciiDoc-style language that compiles to DITA in the same way that AsciiDoc compiles to DocBook. I wonder what the syntax looks like and if maintains all the same features...."

But no, all they did was remove a bunch of DITA tags and declare it "lightweight," as if the *number of tags* was the problem. Removing the `<simpara>` and `<informaltable>` tags doesn't fundamentally change DocBook. The mere *presence* of tags disqualifies a markup language from receiving the lightweight moniker.

Sure, OASIS added [MDITA](http://docs.oasis-open.org/dita/LwDITA/v1.0/cnprd01/LwDITA-v1.0-cnprd01.html#what-is-mdita), which allows you to incorporate Markdown content into a larger DITA project, but this feature could just as easily have been a part of the core DITA specification. Support for an additional language does not make DITA itself lightweight any more than adding [raw HTML support](http://docutils.sourceforge.net/docs/ref/rst/directives.html#raw-data-pass-through) to reStructuredText made it XML-based.

I assume no malice on the part of OASIS. Maybe they thought "Simplified DITA" implied that the new specification was somehow less useful. Maybe someone had heard about these newfangled lightweight markup languages and said, "Hey, we're aiming for something streamlined. Let's call it 'lightweight.'" Whatever the thought process, OASIS arrived at a name that muddies the waters of a well-defined term and will continue to cause confusion in the technical writing community.
